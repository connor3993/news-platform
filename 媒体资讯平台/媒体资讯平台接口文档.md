# 媒体资讯平台接口文档

## 1. 基础约定

### 1.1 服务地址

本地默认后端地址：

```text
http://localhost:8080
```

前端开发代理通常会把 `/admin`、`/user`、`/common`、`/ws` 转发到后端。

### 1.2 统一响应格式

所有 HTTP 接口统一返回：

```json
{
  "code": 1,
  "msg": "success",
  "data": {}
}
```

常见含义：

| 字段 | 说明 |
| --- | --- |
| `code` | `1` 表示成功，非 `1` 表示失败 |
| `msg` | 响应消息或错误原因 |
| `data` | 业务数据，列表、对象、分页结果或空 |

### 1.3 登录认证

管理端接口请求头：

```text
token: 管理员登录返回的 JWT
```

用户端接口请求头：

```text
authentication: 用户登录返回的 JWT
```

免登录接口：

| 类型 | 接口 |
| --- | --- |
| 管理端 | `POST /admin/auth/login` |
| 用户端 | `POST /user/auth/register`、`POST /user/auth/login`、`GET /user/category/list`、`GET /user/article/page`、`GET /user/article/hot`、`GET /user/article/{id}`、`POST /user/article/{id}/read` |

### 1.4 分页返回格式

分页接口的 `data` 通常为：

```json
{
  "total": 100,
  "records": []
}
```

## 2. 数据状态说明

### 2.1 文章状态 `status`

| 值 | 含义 |
| --- | --- |
| `0` | 草稿 |
| `1` | 待审核 |
| `2` | 审核通过 |
| `3` | 审核驳回 |
| `4` | 已发布 |
| `5` | 已下架 |

### 2.2 作者类型 `authorType`

| 值 | 含义 |
| --- | --- |
| `admin` | 平台编辑/管理员创建 |
| `user` | 用户投稿 |

### 2.3 用户投票状态 `userVote`

| 值 | 含义 |
| --- | --- |
| `1` | 当前用户已点赞 |
| `-1` | 当前用户已点踩 |
| `null` | 当前用户未投票或未登录 |

点踩规则：如果用户已经点赞，再点踩会扣除 1 个点赞，并增加 1 个点踩。

### 2.4 热度分说明

热度分 `hotScore` 只用于后端隐性排序推荐，前端列表不直接展示。

当前公式大致为：

```text
hotScore = viewCount + likeCount * 4 + commentCount * 3 + favoriteCount * 3 - dislikeCount + recencyScore * 8
```

其中 `recencyScore` 与发布时间有关，发布时间越新，分值越高。

## 4. 管理端接口

管理端除登录外都需要请求头：

```text
token: 管理员 JWT
```

### 4.1 管理员登录

```http
POST /admin/auth/login
Content-Type: application/json
```

```json
{
  "username": "admin",
```

### 5.5 查询启用分类列表

```http
GET /user/category/list
```

返回：`List<CategoryVO>`。

### 5.6 用户端文章分页

```http
GET /user/article/page?page=1&pageSize=10&categoryId=1&keyword=AI&sort=hot
```

查询参数：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `page` | 否 | 页码，默认 `1` |
| `pageSize` | 否 | 每页数量，默认 `10` |
| `categoryId` | 否 | 分类 ID |
| `keyword` | 否 | 标题关键词 |
| `sort` | 否 | `hot` 表示按热度排序，其他值或不传按发布时间排序 |

返回：`PageResult<ArticleVO>`。

只返回 `4 已发布` 的文章。

排序说明：

| `sort` | 说明 |
| --- | --- |
| `latest` 或不传 | 最新排序，后端使用热度分并提高新文章权重 |
| `popular` | 最受欢迎排序，按点赞数、评论数、收藏数综合排序 |

### 5.7 热点文章列表

```http
GET /user/article/hot?sort=latest
```

查询参数：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `sort` | 否 | `latest` 按最新/隐性热度排行，`popular` 按最受欢迎排行。后端兼容 `view` 按阅读量排行、`like` 按点赞量排行 |

返回：`List<ArticleVO>`，默认 Top10。

缓存说明：

- 不同榜单分别缓存到 Redis。
- 缓存 key 形如 `media:article:hot:latest`、`media:article:hot:popular`、`media:article:hot:view`、`media:article:hot:like`。
- 文章阅读、点赞、点踩、发布、下架、修改等操作会清理热点缓存。
- 点赞的最终数据不依赖 Redis，MySQL 中 `news_article.like_count` 保存文章点赞数量，`news_article_vote` 保存用户投票记录，Redis 只用于榜单和文章详情缓存。

### 5.8 用户端文章详情

### 5.12 文章点赞

```http
POST /user/article/{id}/like
```

需要登录。

返回：更新后的 `ArticleVO`。

规则：

- 未投票用户点赞：`likeCount + 1`，`userVote = 1`。
- 已点踩用户改为点赞：`dislikeCount - 1`，`likeCount + 1`，`userVote = 1`。
- 已点赞用户重复点赞：计数不重复增加。

### 5.13 文章点踩

```http
POST /user/article/{id}/dislike
```

需要登录。

返回：更新后的 `ArticleVO`。

规则：

- 未投票用户点踩：`dislikeCount + 1`，`userVote = -1`。
- 已点赞用户改为点踩：`likeCount - 1`，`dislikeCount + 1`，`userVote = -1`。
- 已点踩用户重复点踩：计数不重复增加。

### 5.14 收藏/取消收藏文章

```http
POST /user/article/{id}/favorite
```

需要登录。

返回：更新后的 `ArticleVO`。

规则：

- 未收藏时调用：新增收藏记录，`favoriteCount + 1`，`favorited = true`。
- 已收藏时调用：取消收藏，`favoriteCount - 1`，`favorited = false`。
- 收藏记录保存到 MySQL 的 `news_article_favorite` 表。


可匿名调用。


示例：

```text
ws://localhost:8080/ws/audit/1
```

说明：

- 管理端旧前端可继续使用。
- 后端会把审核、用户投稿、定时任务完成等消息推给在线管理员。

### 7.2 统一通知接口

```text
ws://localhost:8080/ws/notify/{receiverType}/{receiverId}
```

路径参数：

| 参数 | 说明 |
| --- | --- |
| `receiverType` | `admin` 或 `user` |
| `receiverId` | 管理员 ID 或用户 ID |

示例：

```text
ws://localhost:8080/ws/notify/admin/1
ws://localhost:8080/ws/notify/user/1
```

消息格式：

```json
{
  "type": "ARTICLE_APPROVED",
  "title": "稿件审核通过",
  "content": "稿件《用户投稿标题》已审核通过",
  "time": "2026-06-21 18:00:00"
}
```

常见消息类型：

| 类型 | 接收方 | 说明 |
| --- | --- | --- |
| `USER_ARTICLE_SUBMIT` | 管理员 | 用户提交新投稿 |
| `ARTICLE_PENDING` | 管理员 | 稿件提交审核 |
| `ARTICLE_AUDIT` | 管理员 | 管理员完成审核动作 |
| `ARTICLE_APPROVED` | 用户 | 用户投稿审核通过 |
| `ARTICLE_REJECTED` | 用户 | 用户投稿审核驳回 |
| `ARTICLE_PUBLISHED` | 用户 | 用户投稿已发布 |
| `ARTICLE_OFFLINE` | 用户 | 用户投稿被下架 |
| `ARTICLE_UPDATED` | 用户 | 用户投稿被管理员修改 |
| `ARTICLE_DELETED` | 用户 | 用户投稿被管理员删除 |
| `TASK_DAILY_STATS` | 管理员 | 前日阅读数据统计完成 |
| `TASK_HOT_REFRESH` | 管理员 | 热点资讯热度刷新完成 |
| `TASK_CACHE_CLEAR` | 管理员 | 过期热点缓存清理完成 |

## 8. 定时任务相关说明

后端基于 Spring Task 实现自动任务：

| 执行时间 | 方法 | 说明 | 通知 |
| --- | --- | --- | --- |
| 每天 00:10 | `generateYesterdayStats` | 统计昨日阅读、发布、审核数据 | 推送给管理员 |
| 每 10 分钟 | `refreshHotScore` | 重新计算文章热度并刷新热点缓存 | 推送给管理员 |
| 每天 02:00 | `clearExpiredHotCache` | 清理热点缓存 | 推送给管理员 |

## 9. 常用测试流程

### 9.1 用户投稿到发布

1. 用户登录：`POST /user/auth/login`
2. 用户投稿：`POST /user/article`
3. 管理员登录：`POST /admin/auth/login`
4. 管理员查看待审核：`GET /admin/article/page?status=1`
5. 管理员审核通过：`POST /admin/article/{id}/approve`
6. 管理员发布：`POST /admin/article/{id}/publish`
7. 用户端列表查看：`GET /user/article/page`

### 9.2 点赞点踩

1. 用户登录：`POST /user/auth/login`
2. 打开文章详情：`GET /user/article/{id}`
3. 点赞：`POST /user/article/{id}/like`
4. 点踩：`POST /user/article/{id}/dislike`
5. 检查返回：点赞数扣回，点踩数增加，`userVote = -1`

### 9.3 WebSocket 通知

1. 用户端连接：`ws://localhost:8080/ws/notify/user/1`
2. 管理端连接：`ws://localhost:8080/ws/audit/1` 或 `ws://localhost:8080/ws/notify/admin/1`
3. 用户投稿后，管理员收到 `USER_ARTICLE_SUBMIT`
4. 管理员审核/发布/删除用户投稿后，用户收到对应消息
