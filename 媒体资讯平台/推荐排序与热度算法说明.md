# 推荐排序与热度算法说明

## 1. 文档目的

本文用于说明媒体资讯平台中和“算法”有关的业务逻辑，包括：

- 文章阅读量如何增长
- 点赞、点踩、收藏、评论如何计数
- 热度分 `hotScore` 如何计算
- 首页“最新 / 最受欢迎”如何排序
- 热点榜单如何排序
- Redis 在榜单与详情缓存中的作用
- 定时任务如何刷新热度

这些逻辑主要位于：

```text
media-server/src/main/java/com/connor/newsplatform/server/service/impl/ArticleServiceImpl.java
media-server/src/main/java/com/connor/newsplatform/server/task/NewsTask.java
```

## 2. 文章互动指标

每篇文章在 `news_article` 表中维护一组统计字段：

| 字段 | 含义 | 增长来源 |
| --- | --- | --- |
| `view_count` | 阅读量 | 用户进入文章详情后调用阅读接口 |
| `like_count` | 点赞数 | 用户点赞 |
| `dislike_count` | 点踩数 | 用户点踩 |
| `favorite_count` | 收藏数 | 用户收藏 |
| `comment_count` | 评论数 | 用户发表评论 |
| `hot_score` | 隐性热度分 | 根据阅读、互动和时间新鲜度综合计算 |

其中 `hot_score` 不直接展示给用户，主要用于排序和推荐。

## 3. 阅读量计算

用户打开文章详情后，前端会调用：

```http
POST /user/article/{id}/read
```

后端处理逻辑：

1. 校验文章存在且状态为“已发布”。
2. 向 `news_read_record` 插入一条阅读记录。
3. 将文章 `view_count + 1`。
4. 同步更新一次基础 `hot_score`。
5. 清理文章详情缓存和热点榜缓存。

当前阅读接口允许匿名调用。如果用户已登录，阅读记录中的 `user_id` 使用当前用户 ID；如果未登录，则使用 `0`。

阅读接口里的即时更新公式偏轻量：

```text
hotScore =
  viewCount
  + likeCount * 4
  + commentCount * 3
  + favoriteCount * 3
  - dislikeCount
```

它的作用是让阅读量变化后马上影响排序。完整的新鲜度加权会在点赞、收藏、评论等互动后，或者每 10 分钟定时任务刷新时重新计算。

## 4. 点赞与点踩逻辑

点赞接口：

```http
POST /user/article/{id}/like
```

点踩接口：

```http
POST /user/article/{id}/dislike
```

投票记录保存在：

```text
news_article_vote
```

该表用 `(user_id, article_id)` 唯一约束保证一个用户对一篇文章只能有一种投票状态。

### 4.1 点赞规则

| 当前状态 | 用户操作 | 结果 |
| --- | --- | --- |
| 未投票 | 点赞 | `like_count + 1`，`userVote = 1` |
| 已点踩 | 点赞 | `dislike_count - 1`，`like_count + 1`，`userVote = 1` |
| 已点赞 | 点赞 | 不重复增加数量 |

### 4.2 点踩规则

| 当前状态 | 用户操作 | 结果 |
| --- | --- | --- |
| 未投票 | 点踩 | `dislike_count + 1`，`userVote = -1` |
| 已点赞 | 点踩 | `like_count - 1`，`dislike_count + 1`，`userVote = -1` |
| 已点踩 | 点踩 | 不重复增加数量 |

这就是“点踩会扣除一个点赞”的实现方式。

## 5. 收藏逻辑

收藏接口：

```http
POST /user/article/{id}/favorite
```

收藏记录保存在：

```text
news_article_favorite
```

同样使用 `(user_id, article_id)` 唯一约束，保证一个用户不能重复收藏同一篇文章。

### 5.1 收藏/取消收藏规则

| 当前状态 | 用户操作 | 结果 |
| --- | --- | --- |
| 未收藏 | 点击收藏 | 新增收藏记录，`favorite_count + 1`，`favorited = true` |
| 已收藏 | 再次点击收藏 | 删除收藏记录，`favorite_count - 1`，`favorited = false` |

收藏数量会参与“最受欢迎”和热度分计算。

## 6. 评论逻辑

评论接口：

```http
POST /user/article/{id}/comment
```

评论分页接口：

```http
GET /user/article/{id}/comment/page
```

评论数据保存在：

```text
news_article_comment
```

发表评论后：

1. 向评论表插入一条评论。
2. 将文章 `comment_count + 1`。
3. 重新计算该文章 `hot_score`。
4. 清理文章详情缓存和热点榜缓存。

评论数量会显示在文章列表卡片上，也会参与“最受欢迎”和热度分计算。

## 7. 热度分算法

热度分字段：

```text
news_article.hot_score
```

完整热度分公式可概括为：

```text
hotScore =
  viewCount
  + likeCount * 4
  + commentCount * 3
  + favoriteCount * 3
  - dislikeCount
  + recencyScore * 8
```

### 7.1 各指标权重

| 指标 | 权重 | 设计含义 |
| --- | --- | --- |
| 阅读量 | `1` | 基础曝光和点击热度 |
| 点赞数 | `4` | 明确正反馈，权重最高 |
| 评论数 | `3` | 代表讨论度和参与度 |
| 收藏数 | `3` | 代表用户长期兴趣 |
| 点踩数 | `-1` | 负反馈，降低推荐分 |
| 新鲜度 | `8` | 新文章获得额外排序机会 |

### 7.2 新鲜度分 `recencyScore`

代码中按发布时间计算文章距离当前时间的小时数：

```text
hours = 当前时间 - 发布时间
```

然后计算：

```text
recencyScore = max(0, 240 - hours) / 10
```

含义：

- 文章发布越新，`recencyScore` 越高。
- 240 小时约等于 10 天。
- 超过约 10 天后，新鲜度分逐渐变为 0。
- 最终公式中还会乘以 `8`，所以新鲜度对“最新”排序有明显拉动。

举例：

| 发布时间距离当前 | `recencyScore` | 乘以 8 后贡献 |
| --- | --- | --- |
| 1 小时 | `23.9` | `191.2` |
| 24 小时 | `21.6` | `172.8` |
| 120 小时 | `12` | `96` |
| 240 小时及以上 | `0` | `0` |

这能避免老文章因为历史阅读量高而长期霸榜，也能让新文章有曝光机会。

### 7.3 即时更新和定时更新的区别

项目中有两类热度更新：

| 场景 | 使用公式 | 说明 |
| --- | --- | --- |
| 阅读文章 | 基础热度公式 | 立即增加阅读量，快速更新热点缓存 |
| 点赞、点踩、收藏、评论 | 完整热度公式 | 用户强互动发生后，重新计算包含新鲜度的热度 |
| Spring Task 每 10 分钟刷新 | 完整热度公式 | 批量修正所有已发布文章的热度分 |

所以面试时可以说明：阅读量变化会即时影响热度，但最终排序会通过定时任务和强互动操作回到完整加权公式。

## 8. 首页排序逻辑

用户端首页文章分页接口：

```http
GET /user/article/page?sort=latest
GET /user/article/page?sort=popular
```

### 8.1 最新排序 `latest`

前端显示为：

```text
最新
```

后端逻辑：

```text
ORDER BY hot_score DESC, publish_time DESC
```

注意：这里不是单纯按发布时间倒序，而是“带新鲜度权重的热度排序”。

这样设计的原因：

- 新文章会因为新鲜度分得到加权。
- 但完全没人阅读、点赞、评论、收藏的文章不会无限靠前。
- 兼顾“新”和“有互动”的内容。

### 8.2 最受欢迎排序 `popular`

前端显示为：

```text
最受欢迎
```

后端逻辑：

```text
ORDER BY like_count DESC,
         comment_count DESC,
         favorite_count DESC,
         publish_time DESC
```

也就是说：

1. 点赞多的文章优先。
2. 点赞相同，看评论数。
3. 评论数相同，看收藏数。
4. 仍相同，再看发布时间。

它偏向真实用户互动，而不是单纯点击量。

## 9. 热点榜排序逻辑

热点榜接口：

```http
GET /user/article/hot?sort=latest
GET /user/article/hot?sort=popular
```

默认返回 Top 10。

### 9.1 按最新排行

参数：

```text
sort=latest
```

后端实际使用：

```text
ORDER BY hot_score DESC, publish_time DESC
LIMIT 10
```

也就是基于隐性热度分排序，其中新鲜度权重会拉高新文章。

### 9.2 按最受欢迎排行

参数：

```text
sort=popular
```

后端实际使用：

```text
ORDER BY like_count DESC,
         comment_count DESC,
         favorite_count DESC,
         publish_time DESC
LIMIT 10
```

### 9.3 兼容排序

后端还保留了两个兼容参数：

| 参数 | 逻辑 |
| --- | --- |
| `sort=view` | 按阅读量倒序 |
| `sort=like` | 按点赞数倒序 |

当前用户端页面主要展示“最新”和“最受欢迎”，`view`、`like` 可以作为后续扩展或接口测试使用。

## 10. Redis 缓存逻辑

点赞、评论、收藏等核心数据不以 Redis 为准，最终都落 MySQL。

Redis 主要用于缓存：

| 缓存内容 | Key |
| --- | --- |
| 文章详情 | `media:article:detail:{articleId}` |
| 热点榜-最新 | `media:article:hot:latest` |
| 热点榜-最受欢迎 | `media:article:hot:popular` |
| 热点榜-阅读量兼容 | `media:article:hot:view` |
| 热点榜-点赞量兼容 | `media:article:hot:like` |
| 今日看板 | `media:dashboard:today` |

### 10.1 为什么不把点赞直接存在 Redis

当前项目选择 MySQL 作为互动数据的最终存储，原因是：

- 点赞、收藏、评论都是需要长期保存的用户行为。
- 需要保证一个用户对一篇文章不能重复点赞或重复收藏。
- MySQL 可以用唯一索引保证一致性。
- 面试和课程项目中，MySQL 方案更容易解释和验收。

Redis 只作为性能优化：

- 缓存文章详情。
- 缓存热点榜 Top 10。
- 减少高频查询时的数据库压力。

### 10.2 缓存清理时机

以下操作会清理文章详情缓存和热点榜缓存：

- 阅读文章
- 点赞
- 点踩
- 收藏或取消收藏
- 发表评论
- 管理员修改文章
- 管理员发布文章
- 管理员下架文章
- 管理员删除文章
- 定时任务重新计算热度

这样可以保证页面看到的统计数量和排序不会长期滞后。

## 11. 定时刷新逻辑

项目使用 Spring Task 定时刷新热度。

相关类：

```text
NewsTask
```

### 11.1 每 10 分钟刷新热点

定时表达式：

```text
0 */10 * * * ?
```

执行逻辑：

1. 查询所有已发布文章。
2. 按热度公式重新计算 `hot_score`。
3. 删除旧热点榜缓存。
4. 重新加载热点榜。
5. 通过 WebSocket 通知管理员热点缓存已刷新。

### 11.2 每天清理热点缓存

定时表达式：

```text
0 0 2 * * ?
```

每天凌晨 2 点删除热点缓存，让后续访问重新构建榜单。

### 11.3 每天生成昨日统计

定时表达式：

```text
0 10 0 * * ?
```

每天 00:10 统计前一天的数据，用于管理端数据看板。

## 12. 面试讲解口径

如果面试官问“你们的热点排序怎么做的”，可以这样回答：

> 我们不是简单按阅读量排序，而是设计了一个综合热度分。热度分由阅读量、点赞数、评论数、收藏数、点踩数和发布时间新鲜度组成。点赞权重最高，因为它是明确正反馈；评论和收藏权重次之，代表参与度和长期兴趣；点踩作为负反馈扣分；发布时间越新会有额外加权，避免老文章因为历史阅读量高长期霸榜。最终热度分主要用于首页最新推荐和热点榜排序，前端不会直接展示这个分值。

如果面试官问“点赞为什么不用 Redis”，可以这样回答：

> 点赞、收藏、评论属于用户行为数据，必须长期保存，而且要防止重复操作，所以最终落 MySQL。比如点赞表有用户 ID 和文章 ID 的唯一约束，可以保证一个用户对一篇文章只有一种投票状态。Redis 主要用于缓存文章详情和热点榜，提升读取性能。每次点赞、评论、收藏后，我们会清理相关缓存，让下一次查询重新加载最新数据。

如果面试官问“最新和最受欢迎有什么区别”，可以这样回答：

> 最新排序不是纯发布时间排序，而是基于热度分排序，并且提高了发布时间新鲜度的权重，所以新文章会更容易出现在前面。最受欢迎排序更强调用户互动，按点赞数、评论数、收藏数依次排序。前者用于推荐新鲜内容，后者用于展示用户真正喜欢和讨论较多的内容。

## 13. 可优化方向

当前算法适合课程项目和面试展示，后续可以继续优化：

- 引入时间衰减函数，例如指数衰减。
- 给不同分类设置不同权重。
- 对异常刷阅读、刷点赞行为做限流和去重。
- 阅读量可以按用户/IP/时间窗口去重，避免刷新页面重复计数。
- 评论可以增加审核状态和敏感词过滤。
- 点赞、收藏可以先写 Redis，再异步批量落库，提高高并发能力。
- 热榜可以使用 Redis ZSet 存储分值，支持实时排行榜。
