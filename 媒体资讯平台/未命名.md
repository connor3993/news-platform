````
## 一、Mapper 基础使用
Mapper 接口只需继承 BaseMapper<T>，不需要写 XML 和 SQL：
```java
public interface NewsArticleMapper extends BaseMapper<NewsArticle> {}
````

表格

|selectById(id)|主键查单条|查稿件详情、查管理员信息|

## 二、LambdaQueryWrapper 条件构造器（项目里最常用的写法）

java

运行

```
// 查已发布资讯，按分类过滤，按发布时间倒序
new LambdaQueryWrapper<NewsArticle>()
    .eq(NewsArticle::getStatus, ArticleStatus.PUBLISHED)      // WHERE status = 4
    .eq(categoryId != null, NewsArticle::getCategoryId, categoryId)  // 动态条件
    .like(StringUtils.hasText(keyword), NewsArticle::getTitle, keyword) // LIKE '%关键词%'
    .orderByDesc(NewsArticle::getPublishTime);                // ORDER BY publish_time DESC

// 统计今天发布了多少
new LambdaQueryWrapper<NewsArticle>()
    .eq(NewsArticle::getStatus, ArticleStatus.PUBLISHED)
    .between(NewsArticle::getPublishTime, begin, end);        // WHERE publish_time BETWEEN ? AND ?
```

```
       view_count, like_count, hot_score

UPDATE news_article 
SET hot_score = view_count + like_count * 4 + comment_count * 3 
              + favorite_count * 3 - dislike_count
```

## 五、数据库底层数据结构

InnoDB 索引用的是 B+ 树，不是 B 树也不是二叉树。关键特征：

1. 非叶子节点只存 key 不存数据，叶子节点存完整行数据
2. 叶子节点之间用双向链表连接，范围查询快
3. 树高一般 3~4 层，3 层能存约 2000 万行（假设每页 16KB，主键 BIGINT）
4. 主键索引（聚簇索引）：叶子节点存整行数据
5. 二级索引：叶子节点存主键值，查完整行需要回表

### 本项目建的索引（init.sql 里）

sql

```
KEY idx_article_status_publish (status, publish_time)  -- 联合索引，查已发布+按时间排序
KEY idx_article_category (category_id)                  -- 按分类查文章
KEY idx_article_author (author_type, author_id)         -- 按作者查文章
KEY idx_article_hot (hot_score)                         -- 热点排序
```

## 六、面试高频追问

1. 为什么用 B+ 树不用 B 树？
    
    B 树每个节点都存数据，导致非叶子节点能存的 key 更少，树更高，磁盘 IO 次数更多。B+ 树非叶子节点只存 key，扇出更大，树更矮。
    
2. 为什么不用哈希？
    
    哈希只适合等值查询，不支持范围查询和排序。BETWEEN、ORDER BY、> 这些全走不了。
    
3. 联合索引最左匹配？
    
    idx_article_status_publish (status, publish_time) 查 WHERE status = 4 ORDER BY publish_time DESC 能用上索引，但单独查 WHERE publish_time > ? 用不上（跳过了 status）。
    
4. 覆盖索引是什么？
    
    如果查询的列全在索引里，不需要回表。比如 SELECT category_id FROM news_article 如果 category_id 在索引里就不用回表。
