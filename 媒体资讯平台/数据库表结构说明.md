## 媒体资讯平台 - 数据库表结构说明

数据库名称：`news_db`，共 12 张表，覆盖用户认证、内容管理、审核流程、数据统计和系统审计五大业务模块。

---

### 一、用户认证模块

#### 1. admin_user（管理员用户表）

存储后台管理人员的账号信息，用于管理端的登录鉴权和权限控制。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 管理员 ID，自增主键 |
| username | varchar(64), UNIQUE | 登录用户名，唯一 |
| password | varchar(128) | 加密后的密码 |
| name | varchar(64) | 管理员姓名（显示用） |
| phone | varchar(20) | 手机号，可为空 |
| role | varchar(32) | 角色标识，默认 `editor`，如 admin、editor |
| status | tinyint | 账号状态：1=正常，0=禁用 |
| create_time | datetime | 创建时间（AOP 自动填充） |
| update_time | datetime | 更新时间（AOP 自动填充） |
| create_user | bigint | 创建人 ID（AOP 自动填充） |
| update_user | bigint | 更新人 ID（AOP 自动填充） |

管理员登录后，后端返回 JWT token，后续所有管理端请求通过 `token` 请求头携带该 token。

#### 2. app_user（普通用户表）

存储前端用户（读者）的账号信息，用于用户端的注册、登录和个人资料管理。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 用户 ID，自增主键 |
| username | varchar(64), UNIQUE | 登录用户名，唯一 |
| password | varchar(128) | 加密后的密码 |
| nickname | varchar(64) | 用户昵称 |
| avatar | varchar(255) | 头像 URL（OSS 地址） |
| status | tinyint | 账号状态：1=正常，0=禁用 |
| create_time | datetime | 注册时间 |
| update_time | datetime | 更新时间 |

普通用户登录后，后端返回 JWT token，后续所有用户端请求通过 `authentication` 请求头携带该 token。注意管理员和普通用户使用不同的请求头，对应不同的 JWT 拦截器。

### 二、内容管理模块

#### 3. news_article（资讯文章主表）

存储所有资讯文章的核心信息，是整个平台最核心的业务表。文章与正文内容分离存储（正文在 `news_article_content` 表），列表查询时不需要加载正文，提升性能。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 文章 ID，自增主键 |
| title | varchar(128) | 文章标题 |
| summary | varchar(255) | 文章摘要，用于列表展示 |
| cover_url | varchar(255) | 封面图 URL（OSS 地址） |
| category_id | bigint, FK | 所属分类 ID，关联 `news_category.id` |
| author_id | bigint | 作者 ID（管理员 ID 或用户 ID） |
| author_type | varchar(16) | 作者类型：`admin`=管理员，`app_user`=普通用户 |
| status | tinyint | 稿件状态（见下方状态流转说明） |
| publish_time | datetime | 发布时间，仅 status=4 时有值 |
| view_count | bigint | 阅读次数，用户进入详情页时 +1 |
| like_count | bigint | 点赞数 |
| dislike_count | bigint | 点踩数 |
| favorite_count | bigint | 收藏数 |
| comment_count | bigint | 评论数 |
| hot_score | decimal(12,2) | 热度分 = 阅读量 + 新鲜度权重，定时任务每 10 分钟重算 |
| reject_reason | varchar(255) | 驳回原因，审核驳回时填写 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |
| create_user | bigint | 创建人 ID |
| update_user | bigint | 更新人 ID |

**稿件状态流转**：

```
0(草稿) → 1(待审核) → 2(审核通过) → 4(已发布) → 5(已下架)
                      ↘ 3(审核驳回) → 修改后 → 1(待审核)
```

| 状态值 | 含义 | 说明 |
|--------|------|------|
| 0 | 草稿 | 刚创建，未提交审核 |
| 1 | 待审核 | 已提交，等待管理员审核 |
| 2 | 审核通过 | 审核通过，等待发布 |
| 3 | 审核驳回 | 审核未通过，需修改后重新提交 |
| 4 | 已发布 | 已发布上线，用户端可见 |
| 5 | 已下架 | 已下架，用户端不可见 |

用户端只能看到 status=4（已发布）的文章。

#### 4. news_article_content（资讯文章正文表）

存储文章的富文本正文内容（HTML 格式），与主表一对一关联。采用分表设计的原因是正文内容通常为 LONGTEXT 级别的大文本，列表查询时加载会严重影响性能。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 自增主键 |
| article_id | bigint, UNIQUE | 关联文章 ID，与 `news_article.id` 一对一 |
| content | longtext | 文章正文 HTML 内容 |

查询文章详情时通过 `article_id` JOIN 获取正文；查询文章列表时只查主表，不加载正文。

#### 5. news_category（资讯分类表）

存储资讯的分类信息，支持管理员进行增删改查和启用/禁用操作。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 分类 ID，自增主键 |
| name | varchar(64), UNIQUE | 分类名称，如"科技""财经" |
| sort | int | 排序权重，数值越小越靠前 |
| status | tinyint | 状态：1=启用，0=禁用 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |
| create_user | bigint | 创建人 ID |
| update_user | bigint | 更新人 ID |

用户端只展示 status=1（启用）的分类。分类列表会缓存到 Redis（key: `media:category:list`，TTL 24 小时），采用 Cache Aside 策略，增删改时删除缓存。

### 三、用户互动模块

#### 6. news_article_vote（文章投票表）

记录用户对文章的点赞和点踩操作。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 自增主键 |
| user_id | bigint | 投票用户 ID，关联 `app_user.id` |
| article_id | bigint | 文章 ID，关联 `news_article.id` |
| vote_type | tinyint | 投票类型：1=点赞，0=点踩 |
| create_time | datetime | 投票时间 |
| update_time | datetime | 更新时间（用户修改投票时更新） |

投票数据会同步更新到 `news_article` 表的 `like_count` 和 `dislike_count` 字段。

#### 7. news_article_favorite（文章收藏表）

记录用户的文章收藏行为。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 自增主键 |
| user_id | bigint | 收藏用户 ID，关联 `app_user.id` |
| article_id | bigint | 文章 ID，关联 `news_article.id` |
| create_time | datetime | 收藏时间 |

收藏数据会同步更新到 `news_article` 表的 `favorite_count` 字段。

#### 8. news_article_comment（文章评论表）

存储用户对文章发表的评论。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 评论 ID，自增主键 |
| article_id | bigint | 所属文章 ID，关联 `news_article.id` |
| user_id | bigint | 评论用户 ID，关联 `app_user.id` |
| content | varchar(500) | 评论内容，最多 500 字 |
| status | tinyint | 评论状态：1=正常，0=已屏蔽 |
| create_time | datetime | 评论时间 |

评论数据会同步更新到 `news_article` 表的 `comment_count` 字段。评论也受内容审核管控，违规评论可被管理员屏蔽。

---

### 四、审核与阅读记录模块

#### 9. news_audit_record（稿件审核记录表）

记录每一次稿件审核操作的详细信息，用于审核流程追溯和管理端审核工作台展示。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 审核记录 ID，自增主键 |
| article_id | bigint | 被审核的文章 ID，关联 `news_article.id` |
| auditor_id | bigint | 审核人 ID，关联 `admin_user.id` |
| audit_status | tinyint | 审核结果：2=通过，3=驳回 |
| audit_comment | varchar(255) | 审核意见，驳回时必填驳回原因 |
| create_time | datetime | 审核时间 |

每次审核操作（通过或驳回）都会写入一条记录，形成完整的审核链路。审核完成后后端会通过 WebSocket 向在线管理员推送审核状态变更通知。

#### 10. news_read_record（阅读记录表）

记录用户的文章阅读行为，用于阅读数据统计和热度计算。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 自增主键 |
| user_id | bigint | 阅读用户 ID |
| article_id | bigint | 阅读的文章 ID，关联 `news_article.id` |
| create_time | datetime | 阅读时间 |

用户进入文章详情页时，前端调用 `POST /user/article/{id}/read` 接口，后端同时做两件事：往本表插入一条阅读记录（用于统计），以及对 `news_article` 表的 `view_count` +1（用于展示和热度计算）。

---

### 五、数据统计与系统审计模块

#### 11. news_daily_stats（每日统计表）

存储平台每日的核心运营数据，用于管理端数据看板的趋势图表展示。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 自增主键 |
| stat_date | date, UNIQUE | 统计日期，每天一条记录 |
| article_count | int | 当日文章总数 |
| publish_count | int | 当日新发布文章数 |
| view_count | bigint | 当日总阅读量 |
| audit_count | int | 当日审核通过数 |
| reject_count | int | 当日审核驳回数 |

数据由定时任务在每天凌晨 00:10 自动统计前一天的数据并写入。管理端数据看板的阅读趋势和发布趋势图表均基于本表的 `begin` 到 `end` 日期范围查询。今日实时数据直接从业务表计算，缓存在 Redis 中（TTL 5 分钟）。

#### 12. sys_operation_log（系统操作日志表）

记录管理端所有操作的审计日志，由 AOP 切面自动采集，用于安全审计和操作追溯。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint, PK | 自增主键 |
| operator_id | bigint | 操作人 ID |
| operator_type | varchar(16) | 操作人类型：admin / app_user |
| operation | varchar(128) | 操作名称，如"新增文章""审核通过" |
| request_uri | varchar(255) | 请求路径，如 `/admin/article` |
| request_method | varchar(16) | 请求方法，如 POST、PUT、DELETE |
| request_params | text | 请求参数（JSON 格式） |
| ip | varchar(64) | 操作者 IP 地址 |
| create_time | datetime | 操作时间 |

通过在 Controller 方法上标注 `@LogRecord` 注解触发日志采集，AOP 切面自动记录操作人、请求信息、IP 和时间。管理端审计日志页面可按操作名称和时间范围查询。

### 表关系总览

```
admin_user ──1:N──→ news_article（author_id, author_type='admin'）
admin_user ──1:N──→ news_audit_record（auditor_id）
admin_user ──1:N──→ sys_operation_log（operator_id, operator_type='admin'）

app_user ──1:N──→ news_article（author_id, author_type='app_user'）
app_user ──1:N──→ news_article_vote（user_id）
app_user ──1:N──→ news_article_favorite（user_id）
app_user ──1:N──→ news_article_comment（user_id）
app_user ──1:N──→ news_read_record（user_id）

news_category ──1:N──→ news_article（category_id）

news_article ──1:1──→ news_article_content（article_id）
news_article ──1:N──→ news_article_vote（article_id）
news_article ──1:N──→ news_article_favorite（article_id）
news_article ──1:N──→ news_article_comment（article_id）
news_article ──1:N──→ news_audit_record（article_id）
news_article ──1:N──→ news_read_record（article_id）

news_daily_stats（独立统计表，由定时任务生成）
sys_operation_log（独立审计表，由 AOP 自动生成）
```
