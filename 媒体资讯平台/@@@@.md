一句话开场（背下来）：

  

> 媒体资讯平台后端，Java 21 + Spring Boot 3，管理端做稿件审核发布，用户端做资讯浏览互动。核心技术：JWT 多端鉴权、Redis 缓存、AOP 审计日志、热点分算法、WebSocket 实时推送。

---

必须能讲清楚的 6 个核心点：

1. 稿件状态机

草稿(0) → 待审核(1) → 审核通过(2) / 驳回(3) → 发布(4) → 下架(5)。用户端只能看到 status=4 的，这一条 eq(status, 4) 就隔离了生产和分发。

  

2. JWT 多端鉴权

管理端用 token 请求头 + admin secret，用户端用 authentication 请求头 + user secret。JWT payload 里放 userId、userType、username。拦截器校验 userType 和路径匹配，防止用户 token 访问管理端。解析成功后塞进 ThreadLocal，请求结束 afterCompletion 里 remove 防止线程复用串号。

4. ThreadLocal
一个请求从 Controller 到 Service 到 Mapper，很多层都需要当前用户 ID。如果靠方法参数一层层传，每个方法签名都要加 Long userId，很脏。ThreadLocal 让 ID 跟线程绑定，同一个请求里任何地方直接 BaseContext.getCurrentId() 就能拿到。

5. 热点分公式（能口述）

`hotScore = 阅读量 + 点赞×4 + 评论×3 + 收藏×3 - 踩 + 新鲜度×8`
`新鲜度 = max(0, 240-发布小时数) / 10`
10 天内有加分，超过 10 天衰减为 0。定时任务每 10 分钟全量重算。
  

4. Redis 缓存策略（Cache Aside）

读：先查缓存 → 没有就查库回填。写：操作数据库 → 删缓存（不是更新缓存）。四个 key 记住过期时间就行：分类列表 24h、热点 Top10 10min、文章详情 30min（仅游客）、今日看板 5min。

  

5. AOP 两个切面

@LogRecord — @AfterReturning 切面，自动写 sys_operation_log，记录操作人/路径/参数/IP。@AutoFill — @Before 切面，通过反射自动填 createTime/updateTime/createUser/updateUser（目前切面已就绪但没标注使用，手动设置更稳）。

  

6. 文章主表和正文表拆分

news_article 存结构化字段，news_article_content 存 LONGTEXT 正文。列表页不查正文，详情页才查，减少 IO 和网络传输。

  

---

  

高频追问快速答：

为什么多模块？ common（公共能力）、pojo（数据模型）、server（业务）三层拆分，依赖方向清晰，复用成本低。

可选鉴权怎么做？ 用户端资讯接口不强制登录，但带了有效 token 仍解析写入 BaseContext，Service 层据此区分游客和登录用户。

点赞怎么防重复？ news_article_vote 表记录用户+文章+类型，再点同类型不处理，点另一类型则切换，同时更新主表冗余的 like_count/dislike_count。

密码安全？ 注册用 BCrypt，登录兼容明文（判断 $2a$ 前缀），逐步迁移。

项目可优化点？ 阅读量 Redis 计数异步落库、操作日志异步写入、WebSocket 消息持久化、OSS 加文件类型校验。

# MyBatis 在本项目中的用法

项目从 MyBatis Plus 迁移至原生 MyBatis，Mapper 接口 + XML 文件实现 SQL 映射：

```java
@Mapper
public interface NewsArticleMapper {
    List<NewsArticle> selectPage(Page<NewsArticle> page,
                                 @Param("status") Integer status,
                                 @Param("categoryId") Long categoryId,
                                 @Param("keyword") String keyword,
                                 @Param("sortColumn") String sortColumn);

    NewsArticle selectById(@Param("id") Long id);

    List<NewsArticle> selectHotList(@Param("limit") int limit,
                                    @Param("sortColumn") String sortColumn);

    int insert(NewsArticle article);

    int updateById(NewsArticle article);

    int updateViewCount(@Param("id") Long id);

    int selectCount(@Param("status") Integer status,
                    @Param("begin") LocalDateTime begin,
                    @Param("end") LocalDateTime end);
}
```

对应 XML 示例：

```xml
<mapper namespace="com.connor.newsplatform.server.mapper.NewsArticleMapper">

    <insert id="insert" useGeneratedKeys="true" keyProperty="id">
        INSERT INTO news_article (title, summary, cover_url, category_id,
                                  author_id, author_type, status, create_time, update_time)
        VALUES (#{title}, #{summary}, #{coverUrl}, #{categoryId},
                #{authorId}, #{authorType}, #{status}, #{createTime}, #{updateTime})
    </insert>

    <update id="updateViewCount">
        UPDATE news_article
        SET view_count = view_count + 1,
            hot_score = view_count + 1 + like_count * 4
                        + comment_count * 3 + favorite_count * 3 - dislike_count
        WHERE id = #{id}
    </update>

    <select id="selectById" resultType="NewsArticle">
        SELECT * FROM news_article WHERE id = #{id}
    </select>

    <select id="selectPage" resultType="NewsArticle">
        SELECT id, title, summary, cover_url, category_id, status,
               publish_time, view_count, like_count, hot_score
        FROM news_article
        <where>
            <if test="status != null">AND status = #{status}</if>
            <if test="categoryId != null">AND category_id = #{categoryId}</if>
            <if test="keyword != null and keyword != ''">AND title LIKE CONCAT('%', #{keyword}, '%')</if>
        </where>
        ORDER BY
        <choose>
            <when test="sortColumn == 'hot_score'">hot_score DESC</when>
            <otherwise>publish_time DESC</otherwise>
        </choose>
    </select>

</mapper>
```

## PageHelper 分页

分页查询统一使用 PageHelper：

```java
// 开启分页（紧接着的 MyBatis 查询会自动拼接 LIMIT）
PageHelper.startPage(pageNum, pageSize);

// 执行 MyBatis 查询
List<NewsArticle> list = newsArticleMapper.selectPage(status, categoryId, keyword, sortColumn);

// 包装为 PageInfo 获取 total
PageInfo<NewsArticle> pageInfo = new PageInfo<>(list);
long total = pageInfo.getTotal();
```

## 条件查询（替代 LambdaQueryWrapper）

MyBatis 使用 XML 动态 SQL 实现条件查询，等价于之前的 LambdaQueryWrapper：

```xml
<!-- 条件 where 替代 LambdaQueryWrapper -->
<select id="selectByCondition" resultType="NewsArticle">
    SELECT * FROM news_article
    <where>
        <if test="status != null">AND status = #{status}</if>
        <if test="categoryId != null">AND category_id = #{categoryId}</if>
        <if test="keyword != null and keyword != ''">AND title LIKE CONCAT('%', #{keyword}, '%')</if>
        <if test="begin != null">AND publish_time &gt;= #{begin}</if>
        <if test="end != null">AND publish_time &lt;= #{end}</if>
    </where>
    ORDER BY publish_time DESC
</select>
```

# 数据库底层数据结构

InnoDB 索引用的是 **B+ 树**，不是 B 树也不是二叉树。

## 关键特征

- 非叶子节点只存 Key，不存数据
- 叶子节点存完整行数据
- 叶子节点之间使用双向链表连接
- 范围查询效率高
- 树高通常只有 3~4 层
- 主键索引属于聚簇索引
- 二级索引叶子节点存储主键值，需要回表

## 本项目索引设计

```sql
KEY idx_article_status_publish (status, publish_time)

KEY idx_article_category (category_id)

KEY idx_article_author (author_type, author_id)

KEY idx_article_hot (hot_score)
```

索引用途：

| 索引 | 作用 |
|--------|--------|
| idx_article_status_publish | 查已发布文章并按时间排序 |
| idx_article_category | 分类筛选 |
| idx_article_author | 作者维度查询 |
| idx_article_hot | 热点排序 |

# 面试高频追问

## 为什么用 B+ 树不用 B 树？

B 树每个节点都存数据，导致：

- 单个节点存储 Key 数量减少
- 树高度增加
- 磁盘 IO 更多

B+ 树非叶子节点只存 Key，因此：

- 扇出更大
- 树更矮
- 查询更快

## 为什么不用 Hash？

Hash 索引只适合：

```sql
=
IN
```

无法支持：

```sql
>
<
BETWEEN
ORDER BY
```

因此数据库主要采用 B+ 树。

## 联合索引最左匹配原则

索引：

```sql
(status, publish_time)
```

可以命中：

```sql
WHERE status = 4
ORDER BY publish_time DESC
```

无法充分利用：

```sql
WHERE publish_time > ?
```

因为跳过了最左列 `status`。

## 什么是覆盖索引？

如果查询字段全部存在于索引中：

```sql
SELECT category_id
FROM news_article;
```

数据库直接从索引返回结果，不需要回表。

优点：

- 减少一次磁盘 IO
- 查询速度更快
