# 媒体资讯平台后端业务逻辑面试讲解

## 1. 项目一句话介绍

这是一个面向媒体行业的新闻资讯发布平台，后端基于 Java 21、Spring Boot 4、MyBatis Plus、MySQL、Redis、JWT、AOP、Spring Task、OSS 和 WebSocket 实现。

系统分为管理端和用户端两类业务：

- 管理端负责分类维护、稿件创建、稿件审核、发布下架、数据看板、操作日志。
- 用户端负责注册登录、浏览资讯、按分类/关键词查询、查看详情、记录阅读、查看热点资讯。

我在后端重点实现了四条主线：

- 内容生产链路：草稿 -> 提交审核 -> 审核通过/驳回 -> 发布/下架。
- 内容分发链路：分类列表、资讯流、资讯详情、热点排行。
- 平台治理链路：JWT 鉴权、ThreadLocal 用户上下文、AOP 审计日志。
- 数据运营链路：阅读记录、热点分、每日统计、数据看板。

## 2. 后端模块拆分

项目是 Maven 多模块结构：

- `media-common`
  公共能力模块，放统一返回结果、异常、JWT 工具、OSS 工具、ThreadLocal 上下文、AOP 注解和切面。

- `media-pojo`
  数据模型模块，放 Entity、DTO、VO。
  Entity 对应数据库表，DTO 接收请求参数，VO 返回前端需要的数据。

- `media-server`
  核心业务模块，放 Controller、Service、Mapper、配置类、拦截器、定时任务、WebSocket。

这样拆分的好处是：公共基础能力和业务代码解耦，模型层独立，后续如果扩展更多服务或前端接口，复用成本比较低。

## 3. 数据库核心表设计

主要业务表有：

- `admin_user`
  管理端账号，包含用户名、密码、姓名、角色、状态。

- `app_user`
  用户端账号，包含用户名、密码、昵称、头像、状态。

- `news_category`
  资讯分类，包含名称、排序、启用状态。

- `news_article`
  稿件主表，包含标题、摘要、封面、分类、作者、状态、发布时间、阅读量、热度分、驳回原因。

- `news_article_content`
  稿件正文表，正文是 LONGTEXT，和稿件主表一对一。

- `news_audit_record`
  审核记录表，记录审核人、审核状态、审核意见。

- `news_read_record`
  阅读记录表，用户阅读资讯后写入，用于统计阅读量。

- `news_daily_stats`
  每日统计表，用于数据看板趋势展示。

- `sys_operation_log`
  操作日志表，由 AOP 自动记录管理端和用户端关键操作。

面试可以这样讲：

> 稿件主表和正文表拆开，是为了让列表查询更轻。列表页只查标题、摘要、封面、状态、阅读量等字段，不需要每次把 LONGTEXT 正文查出来。只有详情页才查正文。

## 4. 统一响应与异常处理

所有接口统一返回：

```json
{
  "code": 1,
  "msg": "success",
  "data": {}
}
```

后端定义了 `Result<T>` 和 `PageResult<T>`：

- `Result<T>` 统一包装成功/失败结果。
- `PageResult<T>` 统一返回 `total` 和 `records`，兼容前端分页表格。

异常统一由 `GlobalExceptionHandler` 处理，主要覆盖：

- 业务异常 `BusinessException`
- JWT 解析异常或 token 错误
- 参数校验异常
- 文件上传异常
- 数据唯一键冲突
- 未知系统异常

面试可以这样讲：

> Controller 不直接处理 try-catch，而是把业务错误抛成 BusinessException，最终由全局异常处理器统一转换成前端固定格式。这样前端只需要判断 code 是否为 1。

## 5. JWT 多端鉴权逻辑

系统有两套登录端：

- 管理端请求头使用 `token`
- 用户端请求头使用 `authentication`

JWT payload 中放三个字段：

- `userId`
- `userType`
- `username`

其中 `userType` 是 `admin` 或 `user`，用来区分 token 属于哪个端。

登录流程：

1. 前端提交用户名和密码。
2. 后端查询 `admin_user` 或 `app_user`。
3. 校验账号是否存在、密码是否正确、账号是否启用。
4. 生成 JWT。
5. 返回用户基本信息和 token。

鉴权流程：

1. 请求进入 `JwtTokenInterceptor`。
2. 根据请求路径判断是 `/admin/**` 还是 `/user/**`。
3. 管理端从 `token` 请求头取 token，用户端从 `authentication` 请求头取 token。
4. 使用对应的 secret 解析 JWT。
5. 校验 `userType` 是否和当前端匹配。
6. 把 `userId` 和 `userType` 写入 `BaseContext`。
7. 请求结束后在 `afterCompletion` 中清理 ThreadLocal。

面试可以这样讲：

> 我没有简单地用一套 token 混用两个端，而是通过请求头和 userType 区分管理端、用户端，避免用户 token 访问管理端接口。解析成功后把当前用户信息放进 ThreadLocal，Service 层就不用反复传 userId。

## 6. ThreadLocal 用户上下文

`BaseContext` 封装了两个 ThreadLocal：

- 当前用户 ID
- 当前用户类型

核心方法：

- `setCurrentId`
- `getCurrentId`
- `setCurrentType`
- `getCurrentType`
- `removeCurrent`

使用场景：

- 新增稿件时设置 `authorId`
- 新增/修改分类时设置 `createUser`、`updateUser`
- 审核稿件时记录 `auditorId`
- AOP 审计日志记录操作人

面试可强调：

> ThreadLocal 一定要在请求结束后清理，否则 Tomcat 线程池会复用线程，可能出现用户信息串号的问题。所以我在拦截器的 afterCompletion 里调用 removeCurrent。

## 7. 分类业务逻辑

管理端分类功能：

- 分页查询
- 新增分类
- 修改分类
- 删除分类
- 启用/禁用分类

用户端分类功能：

- 只查询启用状态的分类

缓存策略：

- 用户端分类列表缓存到 Redis。
- 缓存 key 是 `media:category:list`。
- 过期时间是 24 小时。
- 管理端新增、修改、删除、启用禁用分类后，删除该缓存。

查询流程：

1. 用户端请求 `/user/category/list`。
2. 先查 Redis。
3. Redis 有数据，直接返回。
4. Redis 没数据，查 MySQL。
5. 将查询结果写回 Redis。

面试可以这样讲：

> 分类是高频读、低频写的数据，非常适合做缓存。写操作后主动删除缓存，下一次读请求再回源数据库并重建缓存，这是一种 Cache Aside 模式。

## 8. 稿件状态流转

稿件状态设计：

| 状态 | 含义 |
| --- | --- |
| 0 | 草稿 |
| 1 | 待审核 |
| 2 | 审核通过 |
| 3 | 审核驳回 |
| 4 | 已发布 |
| 5 | 已下架 |

完整业务流程：

1. 编辑创建稿件，默认状态是草稿。
2. 编辑提交审核，状态变为待审核。
3. 管理员审核：
   - 通过：状态变为审核通过，并写入审核记录。
   - 驳回：状态变为审核驳回，保存驳回原因，并写入审核记录。
4. 审核通过后可以发布，状态变为已发布，设置发布时间。
5. 已发布稿件可以下架，状态变为已下架。

用户端只能看到 `status = 4` 的已发布资讯。

面试可以这样讲：

> 我把稿件生产和内容分发通过状态字段隔离。管理端可以看到全状态稿件，用户端查询时强制加 `status = 4` 条件，保证草稿、待审核、驳回内容不会被分发出去。

## 9. 稿件创建和详情逻辑

新增稿件时：

1. `news_article` 保存标题、摘要、封面、分类、作者、状态、阅读量、热度分等结构化字段。
2. `news_article_content` 保存正文 HTML。

查询列表时：

- 只查 `news_article` 主表。
- 补充分类名称。
- 不返回正文内容。

查询详情时：

- 查 `news_article` 主表。
- 查 `news_article_content` 正文。
- 补充分类名称。

这样做的原因：

> 列表页访问频率高，如果每次都查 LONGTEXT 正文，会增加数据库 IO 和网络传输。正文独立出来后，列表查询更轻，详情页再按需加载正文。

## 10. 用户端资讯分发逻辑

用户端资讯接口支持：

- 最新资讯
- 分类资讯
- 标题关键词搜索
- 热点排序

核心查询条件：

- 必须是已发布稿件：`status = 4`
- 如果传 `categoryId`，按分类过滤。
- 如果传 `keyword`，按标题模糊搜索。
- 如果 `sort = hot`，按热度分排序。
- 默认按发布时间倒序。

面试可以这样讲：

> 用户端内容分发并没有暴露复杂推荐算法，而是实现了新闻资讯平台最基础、最可解释的分发方式：最新、分类、搜索、热点。这些都是前端信息流和热点榜需要的核心能力。

## 11. 阅读记录和热点排行

用户进入详情页后，前端调用：

`POST /user/article/{id}/read`

后端处理：

1. 校验资讯是否存在并且已发布。
2. 写入 `news_read_record`。
3. `news_article.view_count` 加 1。
4. 更新或清理热点缓存。

热点缓存：

- key：`media:article:hot`
- 内容：热点 Top10
- 过期时间：10 分钟

热点分计算思路：

```text
hotScore = viewCount + 发布时间新鲜度权重
```

定时任务每 10 分钟重新计算一次热点分。

面试可以这样讲：

> 热点排行不能每次请求都实时聚合阅读记录，否则用户量上来后数据库压力很大。所以我把阅读行为落表，文章表保留冗余的 viewCount 和 hotScore，再用 Redis 缓存热点 Top10，定时刷新热度分。

## 12. Redis 缓存设计

项目中主要缓存：

| key | 内容 | 过期时间 |
| --- | --- | --- |
| `media:category:list` | 启用分类列表 | 24 小时 |
| `media:article:hot` | 热点资讯 Top10 | 10 分钟 |
| `media:article:detail:{id}` | 已发布资讯详情 | 30 分钟 |
| `media:dashboard:today` | 今日看板数据 | 5 分钟 |

缓存更新策略：

- 分类变更后删除分类缓存。
- 稿件审核、发布、下架后删除热点和详情缓存。
- 阅读量变化后删除热点和详情缓存。
- 看板数据缓存短时间过期，减少统计查询压力。

面试可以这样讲：

> 我采用的是旁路缓存模式，也就是先读缓存，缓存没有再查数据库并回填；写操作不直接更新缓存，而是删除相关缓存，让下一次读自动重建，避免缓存和数据库不一致。

## 13. AOP 审计日志

项目定义了 `@LogRecord("操作名称")` 注解。

加了注解的方法执行成功后，`LogRecordAspect` 会自动记录：

- 操作人 ID
- 操作端类型
- 操作名称
- 请求路径
- 请求方法
- 请求参数
- IP
- 操作时间

日志写入 `sys_operation_log`。

使用场景：

- 新增分类
- 修改分类
- 删除分类
- 新增稿件
- 修改稿件
- 提交审核
- 审核通过/驳回
- 发布/下架稿件
- 修改用户资料
- 文件上传

面试可以这样讲：

> 审计日志是横切关注点，如果每个业务方法里手写日志，会造成大量重复代码。所以我用 AOP 加自定义注解解耦，业务代码只关注业务逻辑，日志由切面统一处理。

## 14. AOP 公共字段自动填充

项目定义了：

- `@AutoFill(OperationType.INSERT)`
- `@AutoFill(OperationType.UPDATE)`

设计目标是自动填充：

- `createTime`
- `updateTime`
- `createUser`
- `updateUser`

当前业务服务中也显式设置了关键公共字段，原因是：

- 启动和联调更直观。
- 对 MyBatis Plus 的方法调用兼容性更稳。
- AOP 基础设施已保留，后续可以进一步将 Mapper 层插入/更新统一改为注解驱动。

面试可以这样讲：

> 公共字段自动填充属于典型的横切逻辑，我实现了注解和切面基础能力。当前核心业务为了稳定也显式设置了公共字段，如果继续优化，可以把 Mapper 的新增/修改方法统一标注 AutoFill，进一步减少 Service 层重复代码。

## 15. 数据看板逻辑

管理端看板包括：

- 今日发布数
- 今日阅读量
- 今日审核数
- 今日驳回数
- 阅读趋势
- 发布趋势
- 热点资讯 Top10
- 分类内容分布

今日看板：

- 从文章表、阅读记录表、审核记录表统计当天数据。
- 使用 Redis 缓存 5 分钟。

趋势数据：

- 从 `news_daily_stats` 查询每日统计。
- 如果某天没有统计数据，返回 0，保证前端图表连续。

定时统计：

- 每天 00:10 统计前一天数据，写入 `news_daily_stats`。

面试可以这样讲：

> 数据看板分实时数据和离线统计两类。今日数据变化频繁，所以短缓存；历史趋势通过每日统计表承载，避免每次图表查询都扫业务明细表。

## 16. Spring Task 定时任务

项目中有三个定时任务：

- 每天 00:10 生成前一天统计数据。
- 每 10 分钟重新计算热点资讯排行。
- 每天 02:00 清理热点缓存。

面试可以这样讲：

> 定时任务主要处理不适合放在用户请求链路里的工作，比如每日统计和热点分重算。这样可以缩短接口响应时间，把批处理任务异步化、周期化。

## 17. OSS 文件上传

通用上传接口：

`POST /common/upload`

请求类型：

`multipart/form-data`

参数名：

`file`

处理流程：

1. 校验文件非空。
2. 生成按年月分组的对象路径。
3. 使用 UUID 生成文件名，避免重名覆盖。
4. 上传到阿里云 OSS。
5. 返回可访问 URL。

面试可以这样讲：

> 图片等静态资源不直接存到应用服务器，也不存数据库，而是上传到 OSS。数据库只保存图片 URL，这样可以降低服务器带宽和存储压力，也方便 CDN 加速。

## 18. WebSocket 审核通知

WebSocket 地址：

`/ws/audit/{adminId}`

管理端登录后可以建立 WebSocket 连接。

当稿件审核通过或驳回时，后端推送消息：

```json
{
  "type": "ARTICLE_AUDIT",
  "title": "稿件审核状态更新",
  "content": "稿件《AI 技术带来媒体生产新变化》已审核通过",
  "time": "2026-06-21 10:00:00"
}
```

面试可以这样讲：

> 审核状态变化是典型的实时通知场景。如果前端轮询接口，会浪费资源且实时性差。所以我用 WebSocket 保存连接，在审核状态变更后主动推送消息。

## 19. 管理端关键接口链路

### 管理员登录

`POST /admin/auth/login`

链路：

Controller -> AuthService -> AdminUserMapper -> JWT 工具 -> 返回 LoginVO

核心点：

- 校验账号密码。
- 校验状态。
- 生成 admin 类型 token。

### 新增稿件

`POST /admin/article`

链路：

Controller -> ArticleService -> NewsArticleMapper + NewsArticleContentMapper

核心点：

- 默认草稿状态。
- 作者取当前登录管理员 ID。
- 主表和正文表分开保存。

### 提交审核

`POST /admin/article/{id}/submit`

核心点：

- 状态从草稿改成待审核。
- 删除相关缓存。

### 审核通过

`POST /admin/article/{id}/approve`

核心点：

- 状态改成审核通过。
- 写审核记录。
- WebSocket 推送审核消息。
- 清理缓存。

### 发布稿件

`POST /admin/article/{id}/publish`

核心点：

- 只有审核通过或已下架稿件可以发布。
- 状态改成已发布。
- 设置发布时间。
- 清理热点和详情缓存。

## 20. 用户端关键接口链路

### 用户登录

`POST /user/auth/login`

核心点：

- 校验用户账号密码。
- 生成 user 类型 token。
- 返回用户信息和 token。

### 查询资讯列表

`GET /user/article/page`

核心点：

- 只查已发布资讯。
- 支持分类、关键词、最新/热点排序。
- 返回分页结构。

### 查询资讯详情

`GET /user/article/{id}`

核心点：

- 只允许查看已发布资讯。
- 优先查 Redis 详情缓存。
- 缓存没有再查数据库。
- 返回正文内容。

### 记录阅读

`POST /user/article/{id}/read`

核心点：

- 写阅读记录。
- 增加文章阅读量。
- 清理详情和热点缓存。

## 21. 面试高频问题回答

### Q1：这个项目你主要负责什么？

可以答：

> 我主要负责后端整体设计和核心业务实现，包括多模块工程搭建、管理端和用户端 REST 接口、JWT 多端鉴权、ThreadLocal 用户上下文、稿件审核发布流程、Redis 缓存、数据看板统计、AOP 审计日志、OSS 上传和 WebSocket 审核通知。

### Q2：为什么要做多模块？

可以答：

> 因为后端既有公共基础能力，也有数据模型和业务服务。如果全部放在一个模块里，后期会比较混乱。我拆成 common、pojo、server 三层，common 放可复用工具和基础设施，pojo 放模型，server 放业务实现，依赖方向更清晰。

### Q3：JWT 是怎么做多端隔离的？

可以答：

> 管理端和用户端使用不同请求头和不同 secret。管理端 token 放在 `token` 请求头，用户端 token 放在 `authentication` 请求头。JWT 里还有 `userType` 字段，拦截器会校验 token 类型和接口路径是否匹配，防止用户端 token 访问管理端接口。

### Q4：为什么用 ThreadLocal？

可以答：

> 请求经过 JWT 拦截器后，后续 Service 层经常需要当前用户 ID，比如创建稿件、审核稿件、写操作日志。如果每个方法都传 userId 会污染方法签名，所以我把当前用户信息放进 ThreadLocal。请求结束后必须 remove，避免线程池复用导致数据串号。

### Q5：分类列表为什么要缓存？

可以答：

> 分类列表访问频率高，但变更频率低，适合缓存。用户端查询分类时先读 Redis，缓存没有再查 MySQL 并回填。管理端修改分类后删除缓存，下一次请求自动重建。

### Q6：热点排行怎么做？

可以答：

> 用户阅读详情后写阅读记录，并增加文章表的 viewCount。热点分 hotScore 综合阅读量和发布时间新鲜度。热点 Top10 存 Redis，定时任务每 10 分钟刷新，避免每次请求都实时聚合数据库。

### Q7：为什么文章正文单独建表？

可以答：

> 正文是 LONGTEXT，体积比较大。列表页只需要标题、摘要、封面、分类、阅读量等信息，如果正文放在主表并频繁查询，会增加 IO 和网络传输。拆表后列表查询更轻，详情页再查正文。

### Q8：审核通过后做了哪些事？

可以答：

> 审核通过会更新稿件状态为审核通过，写入审核记录，清理相关 Redis 缓存，并通过 WebSocket 向管理端推送审核状态消息。后续发布接口会把状态改成已发布并设置发布时间。

### Q9：AOP 用在了哪里？

可以答：

> 主要用在两个地方：一是 `@LogRecord` 记录操作日志，自动采集操作人、接口路径、请求参数和 IP；二是 `@AutoFill` 公共字段自动填充的基础能力，用于减少 createTime、updateTime、createUser、updateUser 这类重复代码。

### Q10：数据看板怎么避免查询压力太大？

可以答：

> 今日数据使用短期 Redis 缓存，避免高频刷新重复统计。历史趋势不直接扫阅读记录，而是每天凌晨生成 daily_stats 统计表，图表查询直接读统计结果。

### Q11：Redis 和数据库一致性怎么保证？

可以答：

> 我使用 Cache Aside 策略。读的时候先查缓存，缓存没有再查数据库并回填。写操作不直接更新缓存，而是删除相关缓存，比如分类变更删除分类缓存，文章发布/下架/阅读后删除热点和详情缓存，这样下一次读会拿到最新数据。

### Q12：项目现在还有哪些可以优化？

可以答：

> 可以继续优化的点有：第一，阅读量更新可以用 Redis 计数再异步批量落库，减少高并发下数据库 update 压力；第二，WebSocket 可以增加用户端审核通知或消息持久化；第三，操作日志可以改成异步写入，降低业务接口耗时；第四，OSS 上传可以增加文件类型、大小和图片压缩校验。

## 22. 面试时的讲解顺序建议

建议按这个顺序讲，逻辑会比较顺：

1. 先说项目定位：媒体资讯平台，分管理端和用户端。
2. 再说架构：Spring Boot 4 + Java 21，多模块拆分。
3. 讲主业务：稿件从草稿到审核、发布、用户浏览。
4. 讲鉴权：JWT 多端隔离 + ThreadLocal。
5. 讲性能：Redis 缓存分类、详情、热点、看板。
6. 讲运营：阅读记录、热点排行、数据看板、定时任务。
7. 讲治理：AOP 审计日志、统一异常、统一响应。
8. 最后讲可优化点，体现你知道项目边界和演进方向。

## 23. 一分钟项目介绍模板

可以直接这样说：

> 我这个项目是一个媒体资讯平台后端，主要服务于新闻稿件生产和内容分发。管理端可以进行分类管理、稿件创建、审核、发布下架和数据看板查看；用户端可以注册登录、浏览资讯、按分类和关键词搜索、查看详情和热点排行。技术上使用 Java 21 和 Spring Boot 4，数据库用 MySQL，ORM 用 MyBatis Plus，Redis 用来缓存分类、热点、详情和看板数据，JWT 做管理端和用户端的多端鉴权，ThreadLocal 保存当前用户上下文，AOP 做操作日志和公共字段填充，Spring Task 做每日统计和热点刷新，OSS 做图片上传，WebSocket 做审核状态实时推送。项目里我重点实现了稿件状态流转、内容分发、缓存一致性、审计日志和数据统计这些核心后端逻辑。
